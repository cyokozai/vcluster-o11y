# 新 blog 章立て (Outline)

## 設計方針

v1（旧 `docs/blog.md`）の課題:
- **検証 → 結果 → 考察の境界が曖昧**：検証 2 の本文中に Beyla の説明が混在、考察に検証データが点在
- **環境構築の手順が本筋を圧迫**：EKS 構築 10 ステップ、helmfile 解説が中盤を占める
- **データ取得方法と結果の対応が薄い**：PromQL と結果数値の対応が読み取りにくい
- **🔴 4 件の問題が構造に起因**：D-1（数値不整合）, C-1（累積カウント混入）, C-2（Pattern C の二重性）, E-3（推測まま）

v2 の設計原則:
1. **検証ごとに「目的 / シナリオ / 測定項目 / 結果」を統一構造**で記述
2. **環境構築は折りたたみ可能な独立章**にし、本筋を阻害しない
3. **Beyla を「検証 2 末尾の挿入」ではなく独立章**として扱う
4. **考察を「定量根拠 → 解釈 → 限界 → 課題」の 4 段階**で構造化
5. **ログ参照を本文に組み込む**（"詳細は `docs/v2/logs/verify1-*.json` を参照"）
